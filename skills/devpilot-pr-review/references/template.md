# Review Templates

Two templates: one for each inline finding, one for the summary body. The body posts once; inline comments post N times, one per anchored finding. Both go out in a single combined POST (see `posting.md`).

## Inline comment template

Every finding tied to a specific line uses this template. Posted as a single GitHub PR review comment anchored to `(path, line)`. Keep each finding to one screen.

```
### [<Severity>] <short title>

**Behavior today on this branch:** <what the code actually does>

**Why that's a problem:** <impact on users / data / operability>

**Suggested change:** <concrete direction>

**Suggested test:** <test function name + package + the path it should cover>  <!-- include ONLY for missing-test findings (graph-injected or Agent A); omit otherwise -->

**Confidence:** <high | medium | low> — <one-line reason if not high>
```

### Field rules

- **`<Severity>`** — `Blocking`, `Should-fix`, `Consider`, or `Nit`. See severity rubric below.
- **No path/line in the body** — the file path and line number are part of the inline-comment metadata; do not repeat them inside the comment text.
- **Length** — ≤ 8 lines of prose. Long architectural arguments belong in the body's "What's working well" / verdict, not in an inline comment.
- **Anchor for cross-cutting findings** — when a finding has no natural single line (e.g. "this PR adds no tests"), anchor to the most representative line and add a one-liner: *"This comment is about the change as a whole; anchored here for visibility."*
- **Language** — matches the PR's language end-to-end. Chinese PR → Chinese comment, including the field labels.
- **Severity and confidence are independent.** A high-severity bug you are moderately sure about is `Severity: Blocking, Confidence: medium`. Low confidence never automatically demotes severity. See `confidence.md` for the 0–100 rubric the fanout uses internally and the high/medium/low label mapping shown to the author.

## Review body template

The body holds only what doesn't belong on a single line: Verdict, TL;DR, Strengths, the sweep summary, inline-finding counts, Open Questions, disclaimer, metadata. Render the skeleton; fill every placeholder.

```
<!-- devpilot-pr-review (devpilot <version>) -->
@<author-handle> — <severity summary> · reviewed at <short-sha>

## PR Review: <title / #number>

### Verdict
**Ready to merge:** <Yes | With fixes | No>
<one-sentence reasoning>

### TL;DR
<1–2 sentences: single most important thing to address (or, if approving, what makes this clean)>

### Strengths (optional — omit heading if empty)
<0–3 bullets, each pointing at a specific line/decision in the diff. Skip generic praise ("code is clean", "good naming"). Zero concrete strengths → omit the whole section.>

### Unknown-Unknowns Sweep
<List only the dimensions that surfaced a real finding. If all five are clean, collapse to a single line:>
> Sweep: no concerns across local pattern fit / blast radius / pitfalls / stale-training / off-the-shelf.

<Otherwise, one bullet per dimension that surfaced something:>
- Local pattern fit: <finding>
- Blast radius: <finding>
- Known pitfalls (incl. security/data/reversibility): <finding>
- Stale-training check: <finding>
- Hand-rolled vs. off-the-shelf: <finding>

### Security / Performance coverage
<n>/<total_security> security required-checks completed · <n>/<total_perf> performance required-checks completed
<one line per `not_applicable` item with the reason; omit the line entirely if all checks were `checked, no_evidence` or `finding_raised`>

### Inline findings (count by severity)
- Blocking: <n>
- Should-fix: <n>
- Consider: <n>
- Nit: <n>

### Open Questions (optional — omit heading if empty)
<only things the code could not answer, one line each>

---

> _Automated review. Code graph / AST facts: <used | partial: <what was missing> | not available>. Verify findings before acting; not a substitute for human review._

<!--
Generated by devpilot-pr-review (devpilot <version>, e.g. v0.12.2)
Model: <model id, e.g. claude-opus-4-7>
Review event: <REQUEST_CHANGES | COMMENT | APPROVE>
Reviewed commit: <full SHA of the head commit at review time>
Inline comments: <total count>
-->
```

### Field rules

- **`<version>`** — resolve via `devpilot --version` at post time. Take the `vX.Y.Z` token. Fall back to `unknown` if the command is unavailable. Same value goes in both the leading marker and the trailing HTML comment.
- **`<author-handle>`** — `gh pr view --json author -q .author.login`. If unavailable, drop the `@<handle> — ` prefix entirely and start the line with the severity summary.
- **`<severity summary>`** — one of three shapes, derived from inline findings:
  - Any Blocking → `<n> blocking, <n> should-fix` (omit zero-count buckets; keep blocking even if 1)
  - Only Should-fix / Consider / Nit → `<counts>; no blockers` (e.g. `2 should-fix, 1 nit; no blockers`)
  - Zero findings → `looks good to merge`
  - Match the PR language. Chinese PR example: `1 处阻断、2 处建议修复`; approve case: `可以合并`.
- **`<short-sha>`** — first 7 chars of the head commit SHA (`gh pr view --json headRefOid -q '.headRefOid[:7]'` or `git rev-parse --short=7 HEAD`). Same commit as the full SHA in the trailing metadata block.
- **Language** — render every section (TL;DR, sweep, counts, what's working well, Open Questions, disclaimer, metadata) in the PR's language. Translate the disclaimer while preserving its meaning: automated, not authoritative, human judgment required.
- **Verdict** — derived from the highest-severity surviving inline finding:
  - Any Blocking → `Ready to merge: No`
  - Only Should-fix / Consider / Nit → `Ready to merge: With fixes`
  - Zero findings → `Ready to merge: Yes`
- **Severity counts** — derived from the inline comments actually posted. Drop zero-count buckets from the list.
- **Empty sections** — `Open Questions` and `Strengths` are omitted-when-empty (drop the heading entirely). `Unknown-Unknowns Sweep` is always present but collapses to the one-line "no concerns" form when all five dimensions are clean. TL;DR, Inline findings counts, disclaimer, and both metadata blocks are always present.
- **AST / code graph note in disclaimer** — fill the bracketed slot honestly based on whether graph preflight facts were available for this review: `used` (full graph available and consulted), `partial: <what was missing>` (e.g. "callers of foo() unresolved"), or `not available` (no graph for this language/repo). Do not omit this slot.
- **Reviewed commit** — `gh pr view --json headRefOid -q .headRefOid` (or `git rev-parse HEAD` in the checked-out PR). Full 40-char SHA, not abbreviated. Lets readers tell whether a later force-push invalidated the review.
- **Review event** — derived from the highest-severity inline finding:
  - Any Blocking → `REQUEST_CHANGES`
  - Only Should-fix / Consider / Nit → `COMMENT`
  - Zero findings → `APPROVE`

For a fully-filled reference (body + multiple inline comments), see `example-review.md`.

## Severity rubric

Severity describes *impact if the finding is real*, independent of confidence.

- **Blocking** — would cause data loss, security regression, outage, or silently wrong behavior in production.
- **Should-fix** — real bug on a reachable code path, missing test for a risky path, or an unhandled pitfall surfaced by the sweep or checklist.
- **Consider** — design or maintainability feedback worth the author's attention.
- **Nit** — style, naming, wording.

Report findings at every severity; downstream readers filter.
